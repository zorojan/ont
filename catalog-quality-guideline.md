# Документ для разработчика: этап 2 каталога ресурсов

## Цель этапа

На этом этапе задача не в увеличении количества ресурсов, а в повышении качества уже собранной базы. В текущем интерфейсе уже используются фасеты по типу ресурса, отрасли, статусу и формату, а также карточки с описанием, тегами и форматами, поэтому следующий логичный шаг — нормализовать данные, унифицировать краткие описания и улучшить поиск [file:1].

Каталог должен отвечать на три базовых вопроса по каждой карточке: что это за ресурс, для чего он нужен и в каком контексте его применять. Это позволит сделать фильтрацию понятной, поиск более точным, а саму библиотеку — удобной для исследования и повторного использования [file:1].

## Обязательные поля

Для каждого ресурса рекомендуется сохранить следующий минимальный набор полей:

- `id`
- `title`
- `slug`
- `resource_type`
- `industries`
- `formats`
- `status`
- `short_description`
- `tags`
- `source_url`
- `publisher`
- `language`
- `country_or_region`
- `coverage_scope`
- `confidence`
- `updated_at`

Такой набор соответствует текущей логике интерфейса, где уже присутствуют фасеты каталога и карточки с кратким описанием ресурса [file:1].

## Нормализация и классификация

Сначала нужно сделать надежную rule-based классификацию, а не сложную AI-модель. Базовая логика должна опираться на уже доступные данные: заголовок, домен источника, ключевые слова, секции документа, формат и краткий фрагмент текста [file:1].

### Основные классификационные поля

| Поле | Значение | Правило |
|---|---|---|
| `resource_type` | Ontology / Vocabulary / Taxonomy / Framework / Standard | Один основной тип |
| `industries` | 1–3 отрасли | Не больше трех главных |
| `formats` | TTL / RDF / OWL / JSONLD / CSV / XLSX / XML / HTML / PDF | Массив значений |
| `status` | Verified / Candidate | Отдельно от типа |
| `country_or_region` | Global / EU / US / UK / AM / etc. | По источнику, описанию, издателю |
| `coverage_scope` | Universal / Domain-specific / Regional | По назначению ресурса |
| `confidence` | High / Medium / Low | Для очереди ревизии |

### Что означает поле «Нация»

Если под «Нация» имеется в виду фильтр по стране, юрисдикции или региону применения ресурса, лучше назвать поле не `nation`, а `country_or_region`, потому что это точнее и гибче. Для каталога онтологий и классификаторов обычно полезнее не этничность, а именно география происхождения или нормативного применения: например `Global`, `EU`, `US`, `UK`, `Armenia` [file:1].

### Рекомендуемая логика заполнения `country_or_region`

- Если ресурс международный и общеотраслевой — `Global`.
- Если издатель европейский регулятор или стандарт для ЕС — `EU`.
- Если ресурс явно относится к США, например NAICS — `US`.
- Если ресурс национальный классификатор или словарь конкретной страны — использовать ISO-код или короткое название страны.
- Если определить нельзя — `Unknown` и понизить `confidence`.

## Short description

У каждой карточки должно быть короткое описание в одном стиле и примерно одинаковой длины. Оно должно объяснять смысл ресурса обычным языком, а не повторять только название [file:1].

### Шаблон описания

`[Что это за ресурс]. [Что он описывает, стандартизирует или классифицирует]. [Где или зачем используется].`

### Примеры

- `Универсальный словарь для структурированной разметки веб-данных. Описывает сущности вроде организаций, товаров, событий и статей. Используется для SEO, публикации и обмена данными.`
- `Отраслевой классификатор видов экономической деятельности. Стандартизирует сектора и категории бизнеса. Полезен для аналитики, каталогизации и сопоставления данных.`
- `Доменная онтология для кибербезопасности. Описывает угрозы, события, артефакты и связи между ними. Применяется в расследованиях, аналитике и обмене threat intelligence.`

### Правила качества описания

- 140–220 символов.
- Без повторения названия один в один.
- Без маркетинговых формулировок.
- С фокусом на назначение и область применения.
- В одном стиле для всего каталога.

## Фильтры каталога

Текущие фасеты уже дают хорошую основу: тип ресурса, отрасль, статус и формат. Их стоит оставить как первичные, а новые фильтры добавлять только там, где они реально повышают качество поиска и навигации [file:1].

### Основные фильтры

- Тип ресурса.
- Отрасль.
- Формат.
- Статус.
- Язык.
- Страна / регион (`country_or_region`).
- Масштаб покрытия (`coverage_scope`).
- Publisher type.
- Уровень применения: production / reference / experimental.

### Вторичные теги

- Use case: SEO, interoperability, data integration, compliance, analytics.
- Data focus: products, organizations, events, healthcare, finance, manufacturing, cybersecurity.
- Audience: developer, analyst, architect, researcher, business.
- Governance: official, community, academic, vendor.

## Умный поиск по описаниям

Нужно добавить не только обычный полнотекстовый поиск, но и легкий «умный поиск» поверх `title`, `short_description`, `tags` и части исходного текста. При этом на втором этапе лучше не строить сложную AI-систему, а совместить стандартный текстовый поиск с базовой семантической логикой [file:1].

### Рекомендуемая архитектура поиска

#### 1. Базовый индекс

Индексировать поля:

- `title`
- `short_description`
- `tags`
- `publisher`
- `resource_type`
- `industries`
- `formats`
- `country_or_region`

#### 2. Веса полей

Рекомендуемые веса:

- `title`: 5
- `tags`: 4
- `short_description`: 4
- `resource_type`: 3
- `industries`: 3
- `publisher`: 2
- `formats`: 2
- `country_or_region`: 2

#### 3. Что должен уметь поиск

- Поиск по названию.
- Поиск по ключевым словам из описания.
- Поиск по синонимам, например `cyber security` → `cybersecurity`.
- Поиск по сокращениям и нормализованным формам: `json-ld` → `jsonld`, `ecommerce` → `e-commerce`.
- Совмещение с фильтрами слева.
- Подсветка совпадений в описании и тегах.

### Практичная реализация без тяжелого AI

На этом этапе лучше использовать такую схему:

1. Нормализация текста: lowercase, удаление лишних символов, лемматизация по возможности.
2. Словарь синонимов и алиасов.
3. Полнотекстовый поиск по индексируемым полям.
4. Дополнительный rerank по совпадениям в `title` и `short_description`.
5. Фильтрация по фасетам после текстового поиска.

### Когда подключать AI

AI стоит подключать как второй слой, а не как основу поиска:

- для генерации `short_description`, если исходный текст плохой;
- для предложения 3–7 дополнительных тегов;
- для определения вероятной отрасли, если rule-based логика не уверена;
- для семантического поиска по эмбеддингам на следующем этапе.

На втором этапе AI не должен быть единственным источником классификации. Он полезен как помощник для нормализации и обогащения, но базовая структура должна собираться из исходных данных и контролируемых словарей [file:1].

## Порядок внедрения

### Шаг 1

Нормализовать справочники значений:

- тип ресурса;
- отрасли;
- форматы;
- статусы;
- страны и регионы;
- базовые теги.

### Шаг 2

Добавить обязательный `short_description` для каждой записи и привести все описания к единому стилю.

### Шаг 3

Реализовать базовый текстовый поиск с весами полей и словарем синонимов.

### Шаг 4

Добавить `confidence` и очередь на ручную ревизию для слабых записей.

### Шаг 5

Подключить AI только для обогащения: краткие описания, нормализация тегов, вторичная категоризация.

## Итоговая рекомендация

На текущем этапе лучший подход — сначала сделать чистую структуру данных, унификацию описаний, страну/регион, фильтры и качественный поиск по описаниям. Более глубокое AI-обогащение и извлечение скрытых таксономий лучше перенести на следующий этап, когда база уже будет стабильной и предсказуемой [file:1].
