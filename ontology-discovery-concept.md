# Ontology Discovery Catalog — Концептуальный документ

**Проект:** Brain Builder  
**Модуль:** Ontology Discovery Catalog  
**Версия:** 1.0  
**Аудитория:** Разработчик, Product Owner

---

## Зачем это нужно

Когда консультант или AI-интервьюер начинает работу с новым клиентом — пивоварней, клиникой, заводом, розничным магазином — нужна отправная точка: какие понятия, сущности и процессы типичны для этого типа бизнеса?

Вместо того чтобы каждый раз придумывать это с нуля, система использует готовые отраслевые модели — **онтологии, словари и таксономии**, которые уже созданы международными организациями (W3C, GS1, HL7, ETSI и другими). Это проверенные структуры знаний, описывающие реальный мир.

Ontology Discovery Catalog — это **поисковый каталог этих моделей**, который помогает:

1. Найти подходящие модели под конкретный тип бизнеса
2. Понять, какие сущности и понятия они описывают
3. Сгенерировать стартовый опросник для первого разговора с клиентом — на понятном, человеческом языке

---

## Что хранится в каталоге

Каждая запись в каталоге — это один semantic resource: онтология, словарь, таксономия или отраслевой стандарт.

Для каждого ресурса система хранит:

- **Название и описание** — оригинальный текст из официального источника, без изменений
- **Тип** — онтология, словарь, таксономия, фреймворк или стандарт
- **Отрасль и домен** — например: Healthcare, Finance, Food & Agriculture, IoT, BIM
- **Ключевые сущности** — список классов и понятий из данной модели (например: `Patient`, `Diagnosis`, `Prescription`)
- **Форматы** — в каких форматах доступна модель (TTL, OWL, JSON-LD, RDF, Excel)
- **Источник и ссылки** — откуда взята модель, ссылка на оригинал
- **Оценка качества** — насколько ресурс живой, актуальный и пригодный к использованию
- **Статус** — verified (проверен), candidate (кандидат), weak (слабый)

> **Важно:** никакой генерации, интерпретации или обогащения через AI при сборе данных. В базу попадает только то, что официально опубликовано в источнике.

## Откуда берутся данные

Система автоматически собирает данные из нескольких открытых источников:

### LOV — Linked Open Vocabularies
`https://lov.linkeddata.es`

Крупнейший курируемый каталог семантических словарей. Содержит 800+ vocabulary с метаданными, описаниями и ссылками на файлы. Предоставляет открытый REST API. Это основной источник.

### W3ID.org
`https://github.com/perma-id/w3id.org`

Репозиторий постоянных URI для семантических проектов. Каждая папка — отдельный проект или онтология с текстовым описанием и ссылками на ресурс. Содержит 1900+ проектов.

### W3C — официальные стандарты
`https://www.w3.org`

Фиксированный список официальных стандартов W3C: RDF, RDFS, OWL, SKOS, DCAT, PROV-O, SSN/SOSA, Organization Ontology и другие. Добавляются вручную как эталонный слой.

### Внешние отраслевые стандарты
Фиксированный список ресурсов, которые добавляются вручную или полуавтоматически:

| Ресурс | Отрасль |
|--------|---------|
| Schema.org | Universal |
| GS1 Web Vocabulary | Supply Chain, Retail, Food |
| APQC Process Classification Framework | Operations (все отрасли) |
| NAICS / NACE | Отраслевая классификация |
| FIBO | Finance, Banking |
| HL7 FHIR | Healthcare |
| SAREF | IoT, Smart Home |
| FoodOn | Food, Agriculture, HoReCa |
| IFC / BOT | BIM, Construction |
| UNSPSC | Procurement, Catalog |

## Как работает сбор данных

Система периодически обходит источники и сохраняет данные в базу.

Для каждого ресурса:

1. Скачивается официальное описание из источника
2. Сохраняется оригинальный текст — название, описание, теги
3. Извлекается список ключевых сущностей (классов) из файла онтологии если он доступен
4. Проверяется доступность ссылок
5. Рассчитывается оценка качества по объективным признакам (ниже)

**Никакого AI при сборе не используется.** Всё, что попадает в базу — это официальная информация из первоисточника.

### Оценка качества (Quality Score)

Число от 0 до 1, рассчитывается автоматически по следующим признакам:

| Признак | Влияние |
|---------|---------|
| Есть описание длиннее 50 символов | + |
| Ссылка на ресурс работает | + |
| Есть файл онтологии (TTL/OWL/RDF/JSON-LD) | + |
| Есть ссылка на GitHub репозиторий | + |
| Есть упоминание лицензии | + |
| Ресурс обновлялся в последние 2 года | + |
| Присутствует в LOV или W3C | + |

На основе score присваивается статус:
- **Verified** — высокое качество, рекомендуется к использованию
- **Candidate** — неплохой ресурс, требует проверки
- **Weak** — мало данных, использовать осторожно

## Как работает каталог (UI)

Пользователь открывает каталог и видит список всех ресурсов.

Доступны фильтры:
- По типу (онтология, словарь, таксономия, стандарт, фреймворк)
- По отрасли / домену
- По формату файла
- По статусу (verified / candidate)
- Полнотекстовый поиск

При открытии карточки ресурса пользователь видит:
- Полное описание
- Список ключевых сущностей (например: `Recipe`, `Ingredient`, `Batch`, `Supplier`)
- Ссылку на оригинальный источник
- Quality score

## Генерация опросника

Генерация опросника происходит **только по явному запросу пользователя** — когда он открыл карточку ресурса и нажал кнопку «Создать опросник».

### Принцип

Из списка сущностей ресурса (например: `Product`, `Recipe`, `Batch`, `Supplier`, `QualityCheck`) система формирует вопросы для первого разговора с клиентом.

Пользователь может:
- Выбрать конкретные сущности из списка (по умолчанию — все)
- Нажать «Создать опросник»
- Получить список вопросов
- Скопировать или экспортировать

### Требования к вопросам

Вопросы должны быть написаны на человеческом языке — для владельца или менеджера малого и среднего бизнеса, не для IT-специалиста и не для академика.

**Правильно:**
> Есть ли у вас рецептуры на каждый продукт? Как они хранятся — в голове, в Excel, в программе?

**Неправильно:**
> Определены ли экземпляры класса `Recipe` с атрибутами `hasIngredient` типа `FoodMaterial`?

Каждый вопрос привязан к конкретной сущности из онтологии. Это важно — связь сохраняется, чтобы ответы клиента потом можно было соотнести с моделью данных.

### Пример — пивоварня (FoodOn + Schema.org)

Сущности: `Recipe`, `Ingredient`, `Batch`, `Supplier`, `QualityCheck`, `Product`, `Order`, `Client`

| Сущность | Вопрос |
|----------|--------|
| Recipe | Есть ли у вас зафиксированные рецептуры? Где они хранятся? |
| Ingredient | Сколько примерно позиций в составе одного продукта? Отслеживаете происхождение сырья? |
| Batch | Ведёте ли вы учёт партий? Можете по номеру партии найти всю историю? |
| Supplier | Сколько у вас поставщиков? Есть ли договоры и карточки на каждого? |
| QualityCheck | Как проверяете качество? Есть журнал проверок? |
| Product | Сколько позиций в ассортименте? Есть ли сезонные или временные позиции? |
| Order | Принимаете заказы? Как фиксируете — устно, в мессенджере, в программе? |
| Client | Есть ли база клиентов? Разделяете опт и розницу? |

## Что НЕ входит в этот модуль

- Создание или редактирование онтологий
- Маппинг онтологий между собой
- Хранение ответов клиента (это задача модуля Company Brain)
- Генерация данных через AI во время сбора или индексации

---

## Итог

Ontology Discovery Catalog — это **библиотека проверенных бизнес-моделей** с поиском и простым инструментом для генерации стартового опросника.

Источники данных → автоматический сбор → чистая база без AI-интерпретации → поиск и фильтрация → выбор ресурсов → генерация человекопонятных вопросов под конкретного клиента.
